# Wholesale Driver App - Architecture & Features

This document provides a detailed overview of the functions, features, and technical architecture of the Wholesale Driver App.

## 1. Overview
The Wholesale Driver App is a dedicated mobile application built for delivery drivers. Its primary purpose is to allow drivers to receive orders from the admin, manage deliveries, report expenses, maintain stock, and share their live location with the admin for real-time tracking.

## 2. Core Features

### 2.1. Authentication (Login)
*   **Mechanism:** Phone Number and OTP (One-Time Password) generated by the admin app.
*   **Behavior:** It's a one-time login mechanism. Drivers stay logged in persistently.
*   **Session Expiry:** Drivers are automatically logged out after 30 days of inactivity (not receiving any orders). A new OTP is required to log back in.

### 2.2. Order Management
*   **Receive Deliveries:** Drivers receive a list of assigned deliveries containing:
    *   Customer Name & Phone Number
    *   Delivery Address
    *   Order Details: Quantity (mandatory), Weight, Total Price.
*   **Order Actions:** Drivers can mark orders as **Done** or **Rejected** based on the delivery outcome.
*   **Edit Order:** Drivers have the flexibility to edit order details like quantity or weight upon delivery.
*   **Real-time Sync:** All changes (Done, Rejected, Edited) are instantly synced and visible on the admin app.
*   **History & Progress:** Includes an order progress bar and a history section to view past completed orders.

### 2.3. Live Location Tracking
*   **Requirement:** Mandatory feature; the app will not function unless location services are enabled.
*   **Functionality:** Shares the driver's live location with the admin for real-time fleet monitoring.
*   **Technical Implementation:** Utilizes foreground services and background location tracking to ensure location is sent even when the app is minimized.

### 2.4. Stock Management
*   **Initial Stock:** Drivers can input the initial amount of stock they are carrying.
*   **Auto-deduction:** As orders are marked as "Done," the corresponding quantity and/or weight is automatically subtracted from the driver's current stock.
*   **Scope:** Optimized for a single item type (no complex multi-item inventory management needed).

### 2.5. Expense Reporting
*   **Purpose:** Allows drivers to report on-the-job expenses directly to the admin.
*   **Required Data:** 
    *   **Image:** Mandatory photo of the receipt/item (must be clicked via the app's camera, no gallery upload).
    *   **Amount:** Numeric value (Price or Quantity).
    *   **Note:** Optional text description.
*   **Categories:** 
    1.  Petrol/Diesel (Amount = Price)
    2.  Repair (Amount = Price)
    3.  Defective item (Amount = Quantity)
    4.  Other (Amount = Price)

## 3. Technical Architecture

### 3.1. Core Frameworks & Libraries
*   **Framework:** React Native with **Expo** (Version ~56). Expo provides a robust managed workflow and native modules out-of-the-box.
*   **Routing:** **Expo Router** is used for file-based navigation and deep linking capabilities, supporting typed routes.
*   **Styling:** **NativeWind** coupled with **Tailwind CSS**. This allows for utility-first styling consistent with modern web development practices.
*   **State Management:** **Zustand** is used for global state management (handling auth state, current orders, stock, etc.).
*   **Data Fetching & Caching:** **React Query (`@tanstack/react-query`)** manages asynchronous server state, caching, and background updates.

### 3.2. Native Modules & Integrations
*   **Location Services:** `expo-location` and `expo-task-manager` handle the intensive task of background and foreground continuous location tracking.
*   **Camera Integration:** `expo-camera` is integrated specifically for capturing expense receipts on the fly.
*   **Local Storage:** `expo-secure-store` is utilized for securely saving authentication tokens and session data.
*   **Auto-Updates (OTA):** `expo-updates` is configured to check for and apply Over-The-Air updates automatically on load, alongside a custom GitHub auto-update mechanism for full releases.

### 3.3. App Configuration (`app.json` highlights)
*   **Permissions:** The app requests comprehensive permissions including `CAMERA`, `ACCESS_FINE_LOCATION`, `ACCESS_BACKGROUND_LOCATION`, and `FOREGROUND_SERVICE` to support its core features.
*   **Background Modes (iOS):** Specifically enabled `location` under `UIBackgroundModes` for continuous iOS tracking.
*   **Foreground Service (Android):** Configured to run location tracking as a foreground service on Android to prevent the OS from killing the process.

### 3.4. UI/UX Design Philosophy
*   **Target Audience:** Designed to be extremely simple and intuitive, described as usable by a "10-12 year old".
*   **Theming:** Adheres to the overarching theme of the Admin App, with a forced `dark` user interface style out of the box.
